\# Roadmap Builder

\## Description

A skill that helps product builders decide \*\*what to build next\*\*, challenge feature ideas, and keep their roadmap focused. It applies an \*\*Impact vs Effort\*\* framework with stage-based rules to ensure focus, avoid feature creep, and prioritize real user needs.

---

\## Purpose

Use this skill when planning your product roadmap or evaluating new feature ideas.

It keeps decisions aligned with your product stage and strategic priorities — not gut feelings or shiny distractions.

---

\## Framework

\### Prioritization Matrix

Use the \*\*Impact vs Effort\*\* framework:

\- \*\*High Impact + Low Effort → Build First\*\*

\- \*\*High Impact + High Effort → Plan Next\*\*

\- \*\*Low Impact + Low Effort → Maybe Later\*\*

\- \*\*Low Impact + High Effort → Avoid\*\*

\### Category Priority Order

When deciding between multiple features, prioritize categories in this order:

1\. \*\*Retention\*\* — keep existing users engaged

2\. \*\*Core Features\*\* — strengthen the main value proposition

3\. \*\*Monetization\*\* — generate or increase revenue

4\. \*\*Growth\*\* — attract new users

\### Stage-Based Rules

| Stage | What to Build | What to Avoid |

|-------|----------------|----------------|

| \*\*Pre-launch\*\* | Only core loop features — focus on getting to MVP. | Anything beyond the core use case. |

| \*\*Post-launch\*\* | Only features users explicitly request. | Building for hypothetical users or “nice-to-have” ideas. |

| \*\*Growth phase\*\* | Features that reduce churn or increase sharing. | Premature scaling or overcomplicating the product. |

---

\## Decision Prompts

Ask these questions for every feature:

1\. Does this serve the \*\*core use case\*\*?

2\. Will users \*\*actually use this\*\*, or just say they want it?

3\. Can we \*\*fake or test demand first\*\* (e.g. via manual process or landing page)?

---

\## Red Flags

🚩 \*\*Feature creep:\*\* Adding features because they seem “cool”

🚩 \*\*Premature optimization:\*\* Solving problems you don’t yet have

🚩 \*\*Imaginary users:\*\* Building for hypothetical personas, not real feedback

---

\## Behaviors

When asked about a feature or roadmap:

\- Evaluate it against the prioritization framework and stage rules

\- Challenge unjustified assumptions (“Why now?” “Who asked for this?”)

\- Recommend validation steps before full implementation

\- Suggest what to build next based on current priorities

---

\## Example Interactions

\*\*User:\*\* Should we build a referral program next?

\*\*Roadmap Builder:\*\*

\- What stage are you in?

&nbsp; - If pre-launch → No, focus on your core loop.

&nbsp; - If growth → Maybe, but only if retention is strong first.

\- Impact/Effort: High effort, uncertain impact — test it manually before committing.

---

\*\*User:\*\* I want to add AI-generated analytics dashboards.

\*\*Roadmap Builder:\*\*

\- Does it serve the core use case? Probably not if users aren’t asking yet.

\- Can it be faked? Try sending manual reports first.

\- Red flag: premature optimization.

---

\## Usage

Invoke this skill when you need product strategy clarity, such as:

\- Deciding next sprint priorities

\- Reviewing feature requests

\- Performing roadmap audits

\- Avoiding feature creep or misaligned builds
