### [22:13] Fixed the rotation/opacity phantom bug across all 3 surfaces (editor + PDF + 3D)   (branch: fix/designer-rotation-opacity · commits: 54cf6b7)
**Synopsis:** Element rotation + opacity were settable in the inspector but
didn't render. Root-caused each surface (systematic-debugging), fixed all
three so editor = print PDF = 3D preview (WYSIWYG), verified live in browser.

**What worked:** Evidence: typecheck:ui EXIT 0; 199 mocha pass (incl. new
rotation-math + rotated-render tests); chrome-devtools live proof on :3010.
Three verified root causes (NOT taken from the prior journal note — re-derived
from code):
- Editor: `transform: rotate()` was on the SAME <Rnd> node react-draggable
  writes `transform: translate()` to → collision. Fix: rotation moved to an
  inner content wrapper (rotates about center; drag/resize stay on the
  axis-aligned frame). Live-proved via DOM: inner wrapper matrix decodes to
  35deg, Rnd node keeps translate(40px,40px)+opacity 0.5, separate nodes.
- PDF (pdf-renderer.ts): never passed rotation OR opacity to pdf-lib. Added
  opacity to every draw call + center-rotation via NEW pure helper
  `_render/transform.ts` (rotateAbout + pdfAngleFromCss). pdf-lib rotates a
  primitive about its own draw ANCHOR (corner/baseline/ellipse-center) in
  y-UP space; to reproduce CSS center-rotation, rotate each anchor about the
  box center by the same angle handed to pdf-lib's `rotate`. CSS is clockwise
  (y-down) → negate for pdf-lib (y-up). Unrotated/opaque path is byte-identical
  (0-angle + opacity-1 no-ops). drawLine has no rotate option → rotate both
  endpoints. Ellipse anchor already IS the center. Table = rotate every cell
  rect+text anchor about the table center → rigid rotation.
- 3D capture (element-frame.ts): rotation was deliberately omitted (opacity
  already applied); added `transform: rotate()` to the frame style.
Verified all three live: 35deg/0.5 headline + 20deg badge identical in editor,
Print PDF, and 3D face.

**What did NOT work (big time-sink lesson):** After editing element-frame.ts
the 3D capture STILL showed no rotation across TWO full browser reloads + a
hard reload (ignoreCache) + a file re-touch. Misleading signal: opacity
rendered but rotation didn't from the SAME `elementFrameStyle` object, which
first looked like the doc was missing rotation — but the offscreen capture
frame had opacity 0.5 (doc HAD the fields) and a passthrough graphic (badge)
also lacked transform. Real cause: the Next dev server (Windows) file-watcher
systematically MISSED the change to this lazily-loaded (`dynamic(ssr:false)`)
chunk module — the editor's main-bundle change (canvas-area) was picked up,
but the lazy design-preview-3d chunk kept serving a stale compiled
element-frame. Neither browser reload nor a source re-touch fixed it; only a
dev-server RESTART (kill PID on :3010, `npm run dev` fresh) served current
source. LESSON: when a code change provably on disk doesn't appear in a
lazily-loaded Next chunk on Windows, suspect the dev-server module cache /
file-watcher, not your code — restart the dev server before doubting the fix.
